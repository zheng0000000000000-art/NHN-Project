# 미지의 경매장 2주 제작 인원 비교

## 목표

AI 병렬 작업과 Team Loop를 선택적으로 사용해 2주 안에 외부 플레이테스트가 가능한 MVP 후보를 만든다.

공통 완료 범위:

- 물건 12개 전체 카탈로그
- 플레이어 1명과 봇 3명
- 제한 자금과 경매 완주
- 히든 세트 3개와 실제 가치
- 생성 콘텐츠와 안전 프리셋
- 물건 이미지 12개와 기본 UI
- 결과 공개와 Seed 재현
- 핵심 자동 테스트

## 비교

| 항목 | 사람 2명 + AI | 사람 3명 + AI |
|---|---|---|
| 역할 구성 | 제품·게임 / UI·콘텐츠 | 제품·게임 / UI·콘텐츠 / 검증·통합 |
| AI 작업 병렬 수 | 안정적으로 3~4개 | 안정적으로 5~6개 |
| 사람별 예상 투입 | 45~60시간 | 40~60시간 |
| 검증 담당 | 두 사람이 나눠 수행 | 세 번째 사람이 전담 |
| 통합 속도 | 검토 대기 가능성 높음 | 완료되는 작업부터 지속 통합 가능 |
| 플레이테스트 | 내부 4회 전후 | 내부 6회 이상 가능 |
| 에셋 검토 | UI 담당자에게 집중 | 검증 담당자가 규격·누락 검사 지원 |
| 하네스 오탐 처리 | 개발·콘텐츠 작업을 중단시킬 수 있음 | 검증 담당자가 분리 처리 가능 |
| 일정 변경 대응 | 범위 축소가 자주 필요 | 하루 정도의 문제를 흡수할 여지가 있음 |
| 2주 성공 가능성 | 중간~중상 | 높음 |

## 2명 구성

### 역할

- 사람 A: 제품 방향, 게임 규칙, 알고리즘, 통합
- 사람 B: UI, 콘텐츠, 에셋, 접근성

두 사람이 테스트, 실패 재현, Team Loop 작업 확인과 플레이테스트도 나눠 맡는다.

### 장점

- 의사소통과 최종 결정이 빠르다.
- 역할 경계가 단순하다.
- 관리 비용이 비교적 낮다.

### 위험

- AI 결과가 동시에 나오면 검토 대기가 발생한다.
- 통합과 테스트 때문에 제품·UI 판단 시간이 줄어든다.
- 한 사람이 지연되면 해당 영역 전체가 멈출 수 있다.
- 2주 차에는 기능보다 버그와 통합에 시간이 집중될 가능성이 높다.

### 필요한 조건

- 첫날 공통 계약을 확정한다.
- Team Loop는 핵심 코드와 통합 작업에만 사용한다.
- 5일 차에 LLM 없는 완주형 프로토타입을 동결한다.
- 히스토리, 복잡한 필터와 추가 에셋을 우선 제외한다.

## 3명 구성

### 역할

- 사람 A: 제품 방향, 게임 규칙, 알고리즘과 통합 승인
- 사람 B: UI, 콘텐츠, 에셋과 접근성 승인
- 사람 C: 검증, 실패 재현, Team Loop 운영, 플레이테스트와 배포

### 장점

- AI 작업 결과가 검토 대기 상태로 쌓이는 문제를 줄인다.
- A와 B가 제품 판단에 집중하는 동안 C가 테스트와 통합을 유지한다.
- 하네스 오탐과 실제 산출물 실패를 빠르게 구분할 수 있다.
- 매일 플레이 가능한 통합 빌드를 유지하기 쉽다.

### 위험

- 세 번째 사람까지 기능 제작에 투입하면 파일 충돌과 관리 비용이 증가한다.
- 세 사람의 결정 권한이 불명확하면 오히려 승인 시간이 길어진다.
- 범위가 넉넉해 보인다는 이유로 기능을 추가하면 2주 목표를 놓칠 수 있다.

### 필요한 조건

- C는 신규 기능보다 검증과 통합을 우선한다.
- 공통 계약 변경 권한은 A에게 둔다.
- UI와 에셋의 최종 선택 권한은 B에게 둔다.
- 7일 차 이후 대규모 기능·에셋 추가를 금지한다.

## 판단

2명으로도 2주 제작은 가능하지만 결과는 `핵심 플레이를 검증하는 수직 프로토타입`에 가까울 가능성이 높다. 완주 기능을 우선하고 히스토리, 튜토리얼, 추가 프리셋과 상세 결과 분석은 적극적으로 제외해야 한다.

3명 구성은 `외부 플레이테스트 가능한 MVP 후보`까지 도달할 가능성이 더 높다. 차이는 코딩 생산량보다 검증과 통합의 지속성에서 발생한다. AI가 여러 작업을 동시에 완료해도 이를 확인하고 병합할 사람이 없으면 병렬화 효과가 사라지기 때문이다.

따라서 3명이 가능하다면 세 번째 사람을 기능 개발자가 아닌 **검증·통합·플레이테스트 책임자**로 배치하는 안을 권장한다.

## 최종 권장안

```text
기본 운영: 사람 2명
집중 투입: 2주 차에 사람 3명

1주 차
- A: 게임·알고리즘
- B: UI·콘텐츠·에셋

2주 차
- A: 통합 결정과 밸런스
- B: UI·에셋 확정
- C: 검증·플레이테스트·배포
```

세 번째 사람이 전체 2주 동안 참여할 수 있다면 더 안정적이지만, 제한된 참여라면 수직 프로토타입이 완성되는 4~5일 차부터 투입하는 것이 가장 효율적이다.
