# Team Loop 의도와 작업 루프

이 문서는 Team Loop Lite + AI가 어떤 기능을 제공하는지보다, 왜 이런 모양의 도구가 되었는지를 설명한다. 기능 설명은 README와 각 검증 문서를 기준으로 하고, 여기서는 설계 의도, 반복 루프, 필요한 이유, 그리고 이런 생각에 이른 계기를 기록한다.

## 한 줄 의도

Team Loop는 AI와 사람이 함께 일할 때 생기는 불확실한 판단을 작업 보드, 프로그램 검증, 리뷰, 실패 학습으로 나누어 관리하는 로컬 우선 협업 루프다.

AI가 작업을 도울 수는 있지만, 완료 판정까지 AI의 말에 맡기지는 않는다. 대신 작업은 좁은 범위로 나누고, 검증 가능한 것은 하네스로 실행하며, 반복되는 실수는 스킬 또는 하네스로 승격한다.

## 왜 필요한가

AI 코딩 도구를 여러 번 쓰다 보면 같은 문제가 반복된다.

- 작업 지시가 길어질수록 실제로 받은 내용과 의도한 내용이 어긋난다.
- AI는 통과했다고 말하지만, 실제 저장소 상태나 실행 결과는 다를 수 있다.
- 실패가 채팅 안에만 남으면 다음 작업자가 같은 실수를 반복한다.
- 사람은 전체 흐름을 보고 싶지만, 작업 보드는 현재 상태만 보여주기 쉽다.
- 여러 AI 도구를 쓰면 토큰, 한도, 비용, 책임 주체가 흐려진다.

Team Loop는 이 문제를 "AI를 더 믿자"가 아니라 "믿어야 하는 지점을 줄이자"로 접근한다.

## 기본 루프

```text
작업 생성
→ 담당자 또는 AI executor가 작업 가져감
→ 제한된 allowedPaths 안에서 구현
→ 프로그램 검증 실행
→ 실패하면 Failure Case로 기록
→ 실패 이유를 다시 작업자에게 전달
→ 반복 실패는 하네스 또는 스킬로 승격
→ 통과하면 리뷰 요청
→ 사람 또는 별도 reviewer가 승인
→ 완료 기록과 프로젝트 지식으로 남김
```

이 루프에서 중요한 점은 각 단계가 서로 다른 책임을 가진다는 것이다.

- 작업 보드: 누가 무엇을 맡았는지와 현재 상태를 보인다.
- allowedPaths: 작업 범위를 좁혀 충돌과 사고를 줄인다.
- 하네스: 실행 가능한 통과/실패 기준을 담당한다.
- 스킬: 아직 코드로 검증하기 어려운 반복 규칙을 담당한다.
- 리뷰: 하네스가 놓친 제품 의도와 사람 기준을 확인한다.
- WIKI: 끝난 사례, 회의록, 프로젝트 역사를 오래 남긴다.
- 사용량 대시보드: AI 도구 사용 비용과 한도를 보이게 한다.

## 하네스와 스킬을 나눈 이유

모든 실패를 테스트로 만들 수는 없다. 반대로 모든 실패를 문장 규칙으로만 남기면 재발을 막기 어렵다.

그래서 실패를 두 종류로 승격한다.

### 하네스

하네스는 실행 가능한 검증이다.

사용 기준:

- 명령으로 재현할 수 있다.
- 통과/실패가 비교적 명확하다.
- 반복 실행해도 안전하다.
- 작업 완료 여부에 직접적인 신뢰를 더한다.

예:

```text
npm test가 실패했다
→ 해당 명령과 기대 exit code를 가진 회귀 하네스로 만든다
→ 현재 저장소에서 통과해야 활성화한다
```

### 스킬

스킬은 작업자가 다음에 참고해야 하는 절차적 규칙이다.

사용 기준:

- 사람 또는 AI의 습관 문제다.
- 범위 준수, 문서화, 확인 절차처럼 명령 하나로 판정하기 어렵다.
- 그래도 다음 작업 프롬프트에 들어가면 재발 가능성이 줄어든다.

예:

```text
allowedPaths 밖 파일을 수정했다
→ 완료 전 변경 파일 목록을 확인하라는 스킬로 만든다
```

## 자동 학습을 넣은 이유

처음에는 실패 사례를 사람이 보고 하네스나 스킬로 수동 제작하는 흐름이 안전하다. 하지만 작업자가 매번 같은 실패를 보고도 사람이 버튼을 눌러야 한다면, 결국 실패 학습은 밀린다.

그래서 CLI dispatch 루프에는 자동 학습을 붙였다.

```text
검증 실패
→ 실패 사례 기록
→ 작업자가 실패 증거를 다시 받음
→ --auto-learn이면 하네스/스킬 생성 시도
→ 스킬은 활성화 후 현재 작업에 적용
→ 하네스는 작업 통과 후 자체 테스트를 거쳐 활성화
→ 적용 후 다시 검증
```

이 흐름은 "AI가 마음대로 전역 규칙을 만든다"가 아니다. 멤버는 자기에게 배정된 진행 중 작업에서 나온 실패 사례만 자동 학습으로 승격할 수 있고, 실패에서 파생된 본인 산출물만 활성화할 수 있다.

즉 자동화의 범위는 좁게 열고, 검증은 계속 통과해야 한다.

## 리뷰를 남긴 이유

하네스가 통과해도 제품이 좋아졌다고 단정할 수 없다. 특히 게임, UI, 영상 전달력, 심사 기준 같은 영역은 실행 결과만으로 충분하지 않다.

그래서 완료 판정은 두 겹으로 둔다.

```text
하네스 통과
→ 리뷰 요청 가능
→ 별도 reviewer가 승인해야 DONE
```

Solo mode는 예외적으로 존재하지만, 그것도 명시적으로 켜야 하는 모드다. 기본값은 maker와 reviewer를 분리한다.

## 마일스톤과 WIKI를 넣은 이유

작업 보드는 지금 상태를 보여주지만, 시간이 지나면 다음 질문이 남는다.

- 이 작업은 언제 시작했나?
- 언제 가져갔고 언제 끝났나?
- 왜 이 하네스가 생겼나?
- 어떤 실패가 아카이브로 내려갔나?
- 회의에서 어떤 결정을 했나?

마일스톤은 작업의 시간 흐름을 보여주기 위한 장치다. WIKI는 지나간 실패, 회의록, 프로젝트 맥락을 "다시 읽을 수 있는 형태"로 남기기 위한 장치다.

작업이 많아질수록 모든 정보를 한 화면에 펼치는 것은 오히려 방해가 된다. 그래서 WIKI는 평소에는 한 줄 요약으로 접고, 필요할 때만 펼쳐 보는 구조를 택했다.

## 사용량 대시보드를 넣은 이유

AI 협업에서는 작업 품질뿐 아니라 사용량도 운영 조건이다.

보여야 하는 것:

- 누가 얼마나 썼는지
- 어떤 도구와 모델이 많이 쓰이는지
- 외부 도구 한도가 얼마나 남았는지
- 서버 경유 호출과 CLI 직접 사용이 어떻게 다른지

이것은 감시 목적보다 운영 안정성 목적이 크다. 한도와 비용이 보이지 않으면, 중요한 순간에 도구가 멈추거나 특정 사람이 부담을 모르게 떠안을 수 있다.

## 이 생각에 이른 계기

이 프로젝트는 처음부터 거대한 협업 플랫폼을 만들려던 것이 아니다. 출발점은 더 작았다.

AI가 코드를 만들고, 사람이 검토하고, 실패가 반복되는 상황에서 다음 질문들이 계속 생겼다.

- "이 실패를 다음에는 어떻게 안 반복하지?"
- "이걸 그냥 사례 목록에 계속 두는 게 맞나?"
- "하네스로 만들지, 스킬로 만들지 누가 판단하지?"
- "AI가 실패를 봤으면 스스로 다음 작업에 반영할 수 없나?"
- "작업이 많아질수록 사람이 무엇을 먼저 봐야 하지?"
- "아카이브와 회의록도 프로젝트 지식으로 봐야 하지 않나?"

그 질문들의 답이 지금 구조로 이어졌다.

```text
실패 목록
→ 하네스 & 스킬
→ 자동 학습
→ WIKI
→ 마일스톤
→ 사용량과 외부 한도 가시화
```

즉 Team Loop는 단순한 작업 보드가 아니라, AI 협업에서 생기는 실패와 판단을 프로젝트 자산으로 바꾸려는 작은 운영 시스템이다.

## 좋은 사용 방식

- 작업은 작게 만든다.
- allowedPaths는 좁게 잡는다.
- 완료 조건은 사람이 읽어도 명확하게 쓴다.
- 검증 실패는 숨기지 않고 사례로 남긴다.
- 반복 실패는 하네스나 스킬로 승격한다.
- 하네스가 통과해도 리뷰에서 사용자 관점을 본다.
- 회의나 결정은 WIKI에 남긴다.
- 대시보드는 모든 것을 펼치기보다, 자주 보는 것부터 보이게 정리한다.

## 의도적으로 하지 않는 것

- AI의 자기 보고만으로 DONE 처리하지 않는다.
- 실패 사례를 전부 영구 규칙으로 만들지 않는다.
- 모든 판단을 자동화하려고 하지 않는다.
- 작업 보드에 모든 과거 기록을 한꺼번에 펼치지 않는다.
- 전역 Codex 스킬을 무제한으로 자동 생성하지 않는다.

Team Loop 안에서 하네스와 스킬을 돌리는 것은 괜찮다. 그러나 사용자의 전체 개발 환경에 영향을 주는 전역 자동화는 더 강한 승인과 격리가 필요하다.

## 앞으로의 방향

이 문서 기준으로 다음 개선 방향은 명확하다.

- 실패 사례를 자동으로 군집화해 승격 후보를 추천한다.
- 하네스와 스킬의 효과를 통계로 보여준다.
- 리뷰에서 놓친 제품 기준을 WIKI와 스킬로 연결한다.
- 마일스톤에서 작업 지연과 병목을 더 쉽게 볼 수 있게 한다.
- 멤버와 AI worker가 같은 루프를 따르되, 권한과 책임은 더 명확히 보이게 한다.

핵심은 계속 같다.

```text
AI가 더 많이 말하게 하는 것이 아니라,
AI와 사람이 함께 남긴 증거가 다음 작업을 더 안전하게 만들게 한다.
```
