# 브레인스토밍 모드 워크플로우

브레인스토밍 모드는 AI가 바로 구현하거나 제품 명세를 확정하는 기능이 아니다. 기준 문서를 바탕으로 서로 다른 설계 후보를 탐색하고, 비교 근거를 남겨 사람이 선택하기 쉽게 만드는 문서 작업 모드다.

## 기본 흐름

```text
기준 문서 고정
→ 브레인스토밍 요청 작성
→ 독립 후보 생성
→ 후보별 반론과 위험 검토
→ 공통 기준으로 비교
→ 추천안과 차선안 작성
→ 사람의 선택 대기
→ 선택 후 별도 명세·구현 작업
```

## 1. 기준 문서

기준 문서는 제품의 현재 상태와 탐색 범위를 정의한다. 브레인스토밍 실행 중에는 수정하지 않는다.

기준 문서에서는 다음을 구분하는 것이 좋다.

- 불변 조건: 이번 탐색에서도 유지해야 하는 조건
- 현재 설계 가설: 현재안이지만 반론과 근거가 있으면 변경 가능한 조건
- 탐색 가능 영역: 후보가 자유롭게 제안할 수 있는 범위

## 2. 요청서

요청서는 단순한 질문이 아니라 탐색 계약이다. 다음 내용을 포함한다.

- 해결할 설계 질문
- 기준 문서 경로
- 유지해야 할 조건
- 필요한 후보 수와 후보 간 차별성
- 공통 평가 기준
- 후보마다 작성할 장점, 반론, 위험과 검증 방법
- 구현 및 기준 명세 변경 금지 여부

## 3. 독립 후보

각 후보는 다른 후보의 결론을 따라가지 않고 독립적으로 작성한다. 이름만 다르고 핵심 루프가 같은 후보는 하나로 간주한다.

후보 문서에는 최소한 다음을 기록한다.

- 한 문장 정의와 핵심 루프
- 플레이어 또는 사용자가 반복하는 판단
- 기존안에서 달라지는 부분
- 장점과 가장 강한 반론
- 예상 실패 양상
- 자동 검증 방법
- 최소 실험 또는 플레이테스트 기준
- 열린 질문

## 4. 비교와 추천

모든 후보가 작성된 뒤 동일한 기준으로 비교한다. 점수는 근거를 찾기 위한 색인으로만 사용하며 총점만으로 추천을 결정하지 않는다.

결과는 다음을 분리해 기록한다.

- 후보 비교
- 추천안
- 차선안
- 추천안에 대한 가장 강한 반론
- 추천이 뒤집히는 조건
- 아직 결정할 수 없는 질문

추천 문서에서는 제품의 최종 방향과 가장 먼저 수행할 실험을 분리한다.

```markdown
## 제품 방향 판단

- 상태: PROTOTYPE_REQUIRED

## 다음 실험 추천
```

제품 방향 상태는 다음 중 하나를 사용한다.

- `RECOMMENDED`: 현재 근거로 제품 방향을 추천할 수 있음
- `CONDITIONAL`: 특정 조건에서만 추천
- `TIED`: 둘 이상의 후보가 동률
- `INSUFFICIENT_EVIDENCE`: 판단 근거 부족
- `PROTOTYPE_REQUIRED`: 프로토타입 비교 필요
- `RESEARCH_REQUIRED`: 선행 조사 필요

기존 설계 결과가 있다면 정답이 아니라 `baseline` 후보로 비교한다.

## 5. 사람의 선택

AI의 추천은 결정이 아니다. 결정 문서는 기본적으로 `선택 대기` 상태로 생성한다.

사람이 다음 내용을 명시한 뒤에만 후속 작업을 시작한다.

- 선택한 후보
- 선택 이유
- 감수할 위험
- 허용할 명세 변경 범위
- 재검토 조건
- 다음 실험 또는 구현 범위

## 6. 검증과 실패 기록

브레인스토밍 하네스는 다음을 확인한다.

- 기준 문서와 구현 파일을 수정하지 않았는가
- 요구된 수의 후보가 존재하는가
- 후보가 장점만 나열하지 않고 반론과 위험을 포함하는가
- 비교 기준과 추천 근거가 연결되는가
- 차선안과 추천 전환 조건이 존재하는가
- 열린 질문을 억지로 확정하지 않았는가
- Markdown 구조와 변경 범위가 유효한가

검사는 문서 역할에 따라 다르게 적용한다.

- `candidates/*.md`: 기준 명세 영향, 반론, 실패 양상, 검증과 열린 질문
- `comparison.md`: 공통 기준, 명세 충돌, 공통 실패와 미해결 질문
- `recommendation.md`: 제품 방향 상태, 다음 실험, 차선안과 전환 조건
- `decision.md`: 선택 상태, 열린 질문과 재개 조건
- `run.md`: 변경 파일과 검증 기록
- `evidence/**`: 브레인스토밍 내용 검사에서 제외

실패하면 결과를 자동 병합하지 않고 격리 작업공간과 검증 증거를 보존한다.

## 7. 하네스와 스킬 학습

한 번의 실패만으로 새 하네스나 스킬을 만들지 않는다. 여러 실행에서 반복되는지 확인한 뒤 분류한다.

- 출력 형식이나 검사 범위의 반복 문제: 하네스 개선 후보
- AI가 반복해서 놓치는 작업 습관: 스킬 후보
- 특정 문서에만 해당하는 문제: 요청서 또는 기준 문서 수정
- 검사기의 오탐: 학습하지 않고 하네스 결함으로 수정

승격 전에는 실패 원인, 재현 방법, 기대 결과가 최소한 하나의 사례로 남아 있어야 한다.

## 산출물 예시

```text
docs/
├─ 00-product-spec.md
├─ brainstorms/
│  ├─ 001-topic.request.md
│  ├─ 001-topic.baseline.md
│  ├─ candidates/
│  ├─ 001-topic.comparison.md
│  └─ 001-topic.recommendation.md
└─ decisions/
   └─ 001-topic.decision.md
```

핵심 원칙은 아이디어를 많이 만드는 것보다, 서로 다른 선택지를 비교할 수 있는 상태로 보존하고 구현 전에 사람의 결정을 거치는 것이다.
